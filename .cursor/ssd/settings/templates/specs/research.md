# 调研和设计决策模板

---
**目的**：捕获发现结果、架构调查和告知技术设计的理由。

**用法**：
- 在发现阶段记录调研活动和结果。
- 记录对 `design.md` 来说过于详细的设计决策权衡。
- 为未来的审计或重用提供参考和证据。
---

## 摘要
- **功能**: `<feature-name>`
- **发现范围**: 新功能 / 扩展 / 简单添加 / 复杂集成
- **关键发现**:
  - 发现1
  - 发现2
  - 发现3

## 调研日志
记录值得注意的调查步骤及其结果。按主题分组条目以提高可读性。

### [主题或问题]
- **上下文**: 是什么触发了这项调查？
- **咨询的来源**: 链接、文档、API参考、基准测试
- **发现**: 总结见解的简洁要点
- **影响**: 这如何影响架构、契约或实施

_为每个主要主题重复此子章节。_

## 架构模式评估
列出已考虑的候选模式或方法。在有用时使用表格格式。

| Option    | Description                                     | Strengths                       | Risks / Limitations              | Notes                                     |

| Hexagonal | Ports & adapters abstraction around core domain | Clear boundaries, testable core | Requires adapter layer build-out | Aligns with existing steering principle X |

## 设计决策
记录影响 `design.md` 的重大决策。重点关注具有重大权衡的选择。

### Decision: `<Title>`
- **上下文**: 推动决策的问题或需求
- **考虑的替代方案**:
  1. Option A — 简短描述
  2. Option B — 简短描述
- **选定的方法**: 选择了什么以及它如何工作
- **理由**: 为什么这种方法适合当前项目上下文
- **权衡**: 收益vs妥协
- **后续**: 在实施或测试期间要验证的项目

_为每个决策重复此子章节。_
